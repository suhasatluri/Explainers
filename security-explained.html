Floor 5 · Channel security

TLS — the sealed envelope

Transport Layer Security turns a plain conversation between two computers into a sealed, tamper-evident envelope. Anything written inside is unreadable to anyone watching the wire — and the client gets proof it's actually talking to the right server.

What problem does it solve? Without TLS, anything sent over the network — passwords, payment info, your cat photos — is visible to anyone in between (Wi-Fi snoops, ISPs, network gear). TLS gives you three things at once: confidentiality (encrypted), integrity (not tampered with), and server authentication (you reached who you meant to reach).

For the curious — what's actually in the handshake
  • ClientHello — "Hi, here are the TLS versions and cipher suites I support."
  • ServerHello + Certificate — "Here's the one I picked, and here's my X.509 certificate signed by a Certificate Authority you trust."
  • Key exchange — both sides do some math (ECDHE) to agree on a shared secret without ever sending it on the wire.
  • Finished — from now on, every byte is encrypted and authenticated with that shared secret.
  • In modern TLS 1.3 this happens in a single round trip. The client never sent a certificate of its own — the server doesn't know who you are, only that the channel is secure.
Floor 5 · Mutual channel security

mTLS — both sides flash a badge

In regular TLS only the server proves who it is. mTLS (Mutual TLS) adds one more step: the client also presents a certificate, and the server checks it before letting the conversation continue. No valid client cert, no connection.

What problem does it solve? "How do I make sure only my own services talk to this API — not just any caller on the internet?" mTLS gives you machine-to-machine identity baked into the TLS handshake itself, before a single byte of the actual request is processed. It's how most service meshes do service-to-service auth today.

For the curious — the operational reality of running mTLS
  • Every service needs its own certificate, signed by a CA that the other side trusts.
  • Certs expire. Short-lived certs are safer but force you to build rotation. Long-lived certs are easier but risky.
  • Revocation is famously hard. CRLs and OCSP have well-known weaknesses; many places just rely on short expiry.
  • You need a Public Key Infrastructure (PKI) — usually an internal CA, often something like cert-manager, Vault PKI, or SPIFFE/SPIRE.
  • mTLS gives you identity of the caller but not really identity of the human user behind the call, or fine-grained permissions. That's why modern stacks pair it with tokens.
Compare

TLS vs mTLS — at a glance

Same protocol, one extra step, very different use cases.

TLS

🔒 One-way auth

The server proves itself with a certificate. The client stays anonymous at the transport layer.

  • Used by: virtually every HTTPS website on Earth
  • Identifies: the server only
  • Trust model: a public CA your browser already trusts
  • Pain: low. It just works.
mTLS

🔐 Two-way auth

Both server and client present certificates. The server only continues if it recognizes and trusts the client's cert.

  • Used by: service-to-service inside an org, B2B integrations, IoT devices
  • Identifies: both sides
  • Trust model: usually a private/internal CA
  • Pain: PKI ops — issuing, rotating, revoking certs at scale
One sentence summary TLS asks "is this server who it claims to be, and is our chat private?" — mTLS adds "…and is this client also who it claims to be?"
Floors 4 & 3 · Identity vs permission

AuthN vs AuthZ — the bouncer at the door

Two different jobs that get confused all the time. A nightclub bouncer does both, but in order. First they check who you are. Then they check what wristband you're wearing.

🛂

AuthN

"Who are you?"

Proves identity. The bouncer looks at your ID. Either it's you, or it isn't.

Real-world tools • Password + MFA
• TLS / mTLS client cert
• OIDC ID token
• Biometric / passkey
🎟️

AuthZ

"What are you allowed to do?"

Decides permissions. The bouncer looks at your wristband. VIP? Bar only? Backstage?

Real-world tools • OAuth 2.0 scopes
• Roles (RBAC) / attributes (ABAC)
• Policy engines (OPA, Cedar)
• API gateway rules
Why this matters The biggest source of confusion in this whole space: OAuth 2.0 was designed for AuthZ — not AuthN. People kept misusing it for "log in with X", which is why OIDC was created — to put a proper AuthN layer on top.
Floor 2 · The delegation protocol

OAuth 2.0 — the valet key

You drive up to a hotel. You don't hand the valet the deed to your car — you hand them a valet key: starts the engine, opens the doors, but won't open the glovebox and stops working at the end of the night. That's OAuth. A user lets an app do specific things on their behalf, for a limited time, without ever sharing the password.

What problem does it solve? Before OAuth, if a third-party app wanted access to your photos or your email, it had to ask for your username and password — and store them somewhere. Terrifying. OAuth replaces that with: "Click here, log in to the real service, and it'll hand the app a limited access token instead."

Other OAuth flows you'll see in the wild
  • Authorization Code + PKCE — the modern default for any app with a browser or a mobile UI. PKCE prevents code interception.
  • Client Credentials — no human involved. Service-to-service. Client says "here's my ID and secret, give me a token." Common for backend jobs.
  • Device Code — for things that can't show a browser (TVs, CLIs). You see "go to a URL and enter code WXYZ" on another device.
  • Resource Owner Password (legacy) — app collects your password. Discouraged today. Avoid in new designs.
  • Implicit (legacy) — superseded by Auth Code + PKCE. Don't use for new things.
Floor 4 + Floor 2 · Identity on top of OAuth

OIDC — adding "who you are" to OAuth

OpenID Connect is a thin layer built directly on top of OAuth 2.0. Same dance, but you ask for one extra scope — openid — and the Auth Server hands back two tokens instead of one.

access_token

🎟️ The wristband

OAuth. Tells an API what you're allowed to do. Sent on every API call.

  • For: APIs (resource servers)
  • Answers: "is this caller allowed to read /orders?"
  • Can be opaque or a JWT — either way, the API checks it
id_token

🛂 The ID card

OIDC. Tells the client app who the user is. Used once at login.

  • For: the client app itself
  • Answers: "who just signed in?" (name, email, sub)
  • Always a JWT, always signed by the Auth Server
Easy rule of thumb OAuth alone = "this app can do X for you."   OAuth + OIDC = "this app can do X for you, AND here's a signed receipt saying it was definitely you who logged in." Almost every "Sign in with Google / GitHub / Microsoft" button in the world is OIDC under the hood.
Floor 1 · The token format

JWT — the paper the passes are printed on

A JSON Web Token is a format, not a protocol. Three Base64-encoded chunks joined by dots: header.payload.signature. Tap any chunk to see what's inside.

Part 1 · Header
Metadata
{
  "alg": "RS256",
  "typ": "JWT"
}

alg is the signing algorithm. typ is what kind of token this is. That's it — the header just says "here's how to read me."

Part 2 · Payload (the claims)
The actual content
{
  "sub": "123",
  "name": "Jane Doe",
  "iss": "auth.example.com",
  "aud": "orders-api",
  "exp": 1721600000,
  "scope": "orders:read"
}

Each line is a claim. sub=subject, iss=issuer, aud=audience, exp=expiry. Anything extra (scopes, roles, tenant) goes here too.

Part 3 · Signature
The proof
RSA-Sign(
  header + "." + payload,
  authServerPrivateKey
)

Signing the first two parts with the Auth Server's private key. Anyone with the public key can verify it's authentic and untampered — without calling the Auth Server every time.

Signed ≠ encrypted A plain JWT is signed, not encrypted. Anyone who has the token can read its payload — so don't put secrets in it. Treat the payload like a postcard: tamper-proof but readable. If you genuinely need encryption, JWE is a separate spec — but most teams just rely on TLS for confidentiality in transit and short token lifetimes.
How OAuth, OIDC, and JWT actually fit together
  • OAuth 2.0 = the protocol (the conversation between user, client app, auth server, API)
  • OIDC = an extension of OAuth that adds an ID token, so the client knows who logged in
  • JWT = the file format the tokens are often (but not always) printed in
  • OIDC's id_token is always a JWT. OAuth's access_token is often a JWT but is allowed to be anything (e.g. an opaque random string the API looks up via introspection).
Where things live

Certificates vs Tokens — when each one fits

Both are little signed pieces of evidence. The difference is what they're for, how long they live, and how painful they are to rotate.

📜 Certificates (X.509)

Long-lived identity, baked into the TLS layer.
🏭
What identifiesA machine, service, or device — rarely a human user.
⏱️
LifetimeMonths to years (or hours, if you have automation).
🏛️
Trust chainChain back to a Certificate Authority both sides trust.
🛠️
Used byTLS, mTLS, code signing, document signing, SPIFFE/SPIRE.
😬
PainDistribution, rotation, revocation. Needs real PKI ops.

🎟️ Tokens (e.g. JWT)

Short-lived claims, sit above TLS.
🧑
What identifiesA user, a session, or a delegated call — what they're allowed to do right now.
⏱️
LifetimeMinutes to hours. Renewed via refresh tokens.
🏛️
Trust chainSigned by an Auth Server whose public keys you trust.
🛠️
Used byOAuth 2.0, OIDC, API gateways, SPAs, mobile apps.
😌
PainLower. Expire fast, rotation is automatic, no CRL to babysit.
The clearest mental model Certificates answer "is this machine who it claims to be?" Tokens answer "on behalf of which user, with what permissions, for how long, is this call being made?" Modern stacks happily use both — certificates underneath, tokens on top.
The Migration

From mTLS-only to OAuth / OIDC + JWT

mTLS works. So why are we changing? Flip the switch to see the old pattern, then the modern pattern, side by side. The short version: mTLS proves which machine called us. Today we also need to know which user the machine is acting for, what they're allowed to do, and we need to scale that across many services.

Old · mTLS only
←
Modern · OAuth/OIDC + JWT

Old pattern. Every service has its own cert from an internal CA. mTLS proves "this is service-a calling service-b" — but nothing in the call tells service-b which user service-a is acting for, or what that user is allowed to do.

Why mTLS-only struggles at scale
  • No user identity in the call — services have to ask "who's the user?" out-of-band
  • No standard way to express fine-grained permissions (just "you have a valid cert")
  • Cert lifecycle is heavyweight: issuance, rotation, revocation, CA management
  • Hard to delegate: a service can't easily say "I'm acting on behalf of Jane right now"
  • Cross-org integrations turn into "please put your CA on our trust list"
Why modern stacks add OAuth/OIDC + JWT
  • User identity flows from the edge all the way through the call chain (in the JWT)
  • Fine-grained permissions via scopes & claims — easy to express, easy to enforce
  • Short token lifetimes mean revocation is mostly "wait a few minutes"
  • Auth logic lives in one place (the Auth Server), not scattered across services
  • Plays well with humans, SPAs, mobile, partners, and machine-to-machine alike
The migration in practice — what teams actually do
  • Keep TLS everywhere. The pipe still needs to be encrypted. This part doesn't change.
  • Don't necessarily drop mTLS. Many modern designs keep mTLS for service-to-service identity (especially in a service mesh) and add OAuth tokens on top for user identity and authz. Belt and braces.
  • Introduce an Auth Server. Pick one (Keycloak, Okta, Entra ID, Auth0, AWS Cognito, etc.). It becomes the single source of truth for identity.
  • Validate tokens at the edge first. An API gateway can verify JWT signatures, expiry, audience, and scopes before traffic even reaches your service.
  • Propagate the user. Downstream services either re-verify the same JWT or use token exchange to get a downstream-scoped token.
  • Migrate one service at a time. Run mTLS-only and OAuth side-by-side during the cutover. Don't flip everything in one go.

The Last Word

If you remember one thing: TLS locks the pipe, mTLS makes both sides show ID, and everything else — OAuth, OIDC, JWT — is about who the user is and what they're allowed to do inside that pipe.

  • TLS = the conversation is private & the server is real (one-way)
  • mTLS = both sides prove identity with certificates (two-way)
  • AuthN proves who you are; AuthZ decides what you can do
  • OAuth 2.0 is a delegation protocol — it issues access tokens
  • OIDC = OAuth + identity layer — it adds an id token (always a JWT)
  • JWT is a format: header.payload.signature, signed not encrypted
  • Certificates = long-lived machine identity. Tokens = short-lived user/session permissions
  • The migration: keep TLS, often keep mTLS for service identity, add Auth Server + JWTs for user identity & authorization
